# Palantir 风格角色工作台设计

> 日期：2026-05-20  
> 范围：记录 ManuFoundry 从“低代码开发平台”升级为“角色化智能运营工作台”的产品结构、工作流和 UI 设计方向。  
> 参考方向：Palantir Foundry / Gotham 的公开产品理念，重点借鉴“对象中心、事件驱动、关系态势、可执行动作、角色权限”这几类交互思想，而不是复刻其军事或专有界面。

## 1. 核心定位

当前系统已经具备低代码平台能力，包括模型设计、页面配置、表单、菜单、流程、规则、数据源、报表和 AI 助手。

后续产品表达不应只强调“可以搭页面”，而应升级为：

> 用低代码构建业务对象，用本体和图谱连接业务关系，用 AI 分析问题，用流程和动作推动业务闭环。

一句话定位：

> ManuFoundry 是面向制造业的智能运营平台。超级管理员配置系统，业务人员处理业务，管理层看风险和决策，同一套数据通过角色权限呈现为不同工作台。

## 2. 低代码平台放在哪里

低代码平台不是前台主角，而是后台配置能力。

### 2.1 前台：角色工作台

前台面向日常业务用户，回答：

- 今天发生了什么风险？
- 这件事影响了谁？
- 我需要做什么？
- 谁正在处理？
- 处理是否闭环？

典型前台入口：

- 管理驾驶舱
- 质量工作台
- 采购与物料工作台
- 设备维护工作台
- 我的审批
- 智能运营指挥台

### 2.2 后台：平台配置中心

后台面向超级管理员、实施人员、平台管理员，回答：

- 业务对象怎么定义？
- 页面和表单怎么搭？
- 菜单和权限怎么分配？
- 流程和规则怎么配置？
- 数据源和对象关系怎么维护？

典型后台入口：

- 应用管理
- 模型设计
- 表单设计
- 页面设计
- 菜单配置
- 权限配置
- 流程配置
- 规则引擎
- 数据源管理
- 本体关系配置
- 模板市场

## 3. 角色视角

系统应按角色展示不同首页、菜单、数据范围和动作权限。

| 角色 | 默认入口 | 主要关注 | 可执行动作 |
| --- | --- | --- | --- |
| 超级管理员 | 平台配置中心 | 系统搭建、对象建模、权限、流程、规则 | 配置应用、发布页面、配置流程、管理权限 |
| 管理层 | 管理驾驶舱 | 整体风险、交付影响、部门协同、待决策事项 | 查看影响范围、批示、追踪闭环 |
| 质量经理 | 质量工作台 | 质量异常、缺陷原因、CAPA、受影响订单 | 创建整改任务、发起复检、通知负责人 |
| 采购/物料人员 | 采购与料号工作台 | 料号申请、低库存、供应商风险、采购申请 | 提交料号申请、创建采购申请、冻结供应商风险项 |
| 设备工程师 | 设备维护工作台 | 设备健康、预测性维护、维修工单、点检任务 | 创建设备点检、接单、提交维修结果 |
| 审批人 | 我的审批 | 待审批事项、驳回补充、审批记录 | 同意、驳回、转交、评论 |

关键原则：

> 同一个业务事实，不同角色看到不同视图。

例如同一个“3 号产线质量异常”：

- 管理层看到客户交付风险和处理进度。
- 质量经理看到缺陷分布、质检记录和 CAPA。
- 生产主管看到产线、工单和停线建议。
- 采购人员看到供应商和物料批次风险。
- 审批人只看到需要自己处理的审批事项。
- 超级管理员看到该异常规则、流程和页面配置。

## 4. Palantir 风格 UI 思路

这里借鉴的是 Palantir 类产品的结构思想，而不是照搬视觉。

### 4.1 从“菜单表格表单”转向“事件驱动工作台”

传统后台页面通常是：

```text
左侧菜单 -> 中间表格 -> 新增/编辑表单
```

智能运营工作台应更像：

```text
左侧事件队列 -> 中央态势画布 -> 右侧行动面板 -> 底部时间轴
```

### 4.2 推荐页面骨架

```text
┌──────────────────────────────────────────────┐
│ 顶部：全局搜索 / 通知 / 当前角色 / 应用切换   │
├──────────────┬──────────────────┬────────────┤
│ 左侧事件栏    │ 中央态势画布       │ 右侧行动栏 │
│              │                  │            │
│ 质量异常      │ 影响关系图         │ AI 分析    │
│ 供应链风险    │ 工单-设备-物料关系  │ 处置方案   │
│ 设备预警      │ 客户订单影响路径    │ 审批动作   │
│ 待处理任务    │                  │ 所需资源   │
├──────────────┴──────────────────┴────────────┤
│ 底部时间轴：发现异常 -> 分析 -> 派单 -> 审批 -> 闭环 │
└──────────────────────────────────────────────┘
```

### 4.3 区域职责

| UI 区域 | 职责 | 制造业示例 |
| --- | --- | --- |
| 顶部栏 | 全局搜索、通知、用户、应用切换 | 搜索物料、工单、供应商、客户订单 |
| 左侧事件栏 | 当前角色需要关注的事件和任务 | 质量异常、低库存、设备预警、待审批 |
| 中央态势画布 | 展示对象关系、影响路径、业务拓扑 | 供应商 -> 物料 -> 工单 -> 设备 -> 质检 -> 客户订单 |
| 右侧行动栏 | AI 总结、建议动作、审批按钮、相关对象 | 生成整改任务、冻结批次、创建设备点检 |
| 底部时间轴 | 展示计划、过程、审批、处理闭环 | 发现异常、完成分析、创建 CAPA、复检完成 |

## 5. 示例场景：质量异常影响分析与处置

这是第一阶段最适合演示 Palantir 风格的场景。

### 5.1 演示故事

> 今天 3 号产线缺陷率突然升高。系统自动发现风险，关联物料、设备、工单、供应商和客户订单，AI 给出处置建议，质量经理一键生成整改任务，相关负责人收到通知并完成闭环。

### 5.2 页面流程

```text
运营首页
  -> 点击“3 号产线质量异常”
  -> 进入异常详情
  -> 查看影响关系图
  -> 查看 AI 原因分析和建议动作
  -> 点击“生成整改任务”
  -> 自动带出 CAPA 表单
  -> 提交流程并通知负责人
  -> 在时间轴追踪处理状态
```

### 5.3 事件详情页内容

事件摘要：

```text
异常事件：3 号产线缺陷率异常升高
发现时间：2026-05-20 09:18
异常表现：外观划痕缺陷占比上升
关联工单：WO-20260520-003
风险等级：高
```

影响图谱：

```text
供应商 SUP-06
   -> 供应
物料 MAT-21
   -> 用于
工单 WO-20260520-003
   -> 生产于
3 号产线 / 设备 EQ-03
   -> 产生
质检记录 QC-8891
   -> 发现
缺陷 D-102 外观划痕
   -> 影响
客户订单 SO-7788
```

AI 分析：

```text
可能原因：
1. EQ-03 设备振动值连续 3 小时高于正常范围。
2. MAT-21 物料在昨日切换了新批次。
3. 缺陷集中出现在同一工单和同一产线。

影响范围：
1. 影响工单 WO-20260520-003。
2. 可能影响客户订单 SO-7788。
3. 同批物料还有 360 件库存未使用。

建议动作：
1. 暂停 MAT-21 当前批次继续投产。
2. 对 EQ-03 执行设备点检。
3. 创建质量整改任务，由质量经理跟进。
4. 通知生产计划员复核交付风险。
```

业务动作按钮：

- 生成质量整改任务
- 冻结物料批次
- 创建设备点检工单
- 通知相关负责人
- 导出影响分析报告

### 5.4 整改任务表单自动带出

点击“生成质量整改任务”后，系统应弹出任务表单，并自动填入上下文：

| 字段 | 自动值 |
| --- | --- |
| 任务类型 | 质量整改 CAPA |
| 异常来源 | 3 号产线缺陷率异常 |
| 关联工单 | WO-20260520-003 |
| 关联物料 | MAT-21 |
| 关联设备 | EQ-03 |
| 责任人 | 质量经理 |
| 优先级 | 高 |
| 整改建议 | 暂停批次、设备点检、客户订单复检 |

## 6. 业务对象与动作设计

Palantir 风格的重点是对象和动作。对象不是简单表，动作不是简单按钮。

### 6.1 核心业务对象

| 对象 | 说明 |
| --- | --- |
| Supplier | 供应商 |
| Material | 物料 |
| MaterialBatch | 物料批次 |
| WorkOrder | 生产工单 |
| Equipment | 设备 |
| ProductionLine | 产线 |
| Inspection | 质检记录 |
| Defect | 缺陷记录 |
| SalesOrder | 客户订单 |
| RiskEvent | 风险事件 |
| CorrectiveAction | 整改任务 |
| ApprovalTask | 审批任务 |

### 6.2 核心对象关系

```text
Supplier -> supplies -> Material
Material -> has_batch -> MaterialBatch
MaterialBatch -> used_by -> WorkOrder
WorkOrder -> runs_on -> ProductionLine
WorkOrder -> uses -> Equipment
WorkOrder -> produces -> Inspection
Inspection -> finds -> Defect
WorkOrder -> fulfills -> SalesOrder
RiskEvent -> affects -> WorkOrder / MaterialBatch / Equipment / SalesOrder
CorrectiveAction -> resolves -> RiskEvent
```

### 6.3 核心业务动作

| 动作 | 说明 | 权限建议 |
| --- | --- | --- |
| 生成质量整改任务 | 从异常事件创建 CAPA 草稿 | 质量经理 |
| 冻结物料批次 | 暂停风险批次继续投产 | 质量经理、物料负责人 |
| 发起复检 | 对受影响物料或订单创建复检任务 | 质量人员 |
| 创建设备点检工单 | 对异常设备生成点检或维修工单 | 质量经理、设备工程师 |
| 通知相关负责人 | 按事件关系通知质量、生产、采购、计划人员 | 事件负责人 |
| 提交审批 | 将整改、采购或料号申请提交给审批人 | 业务发起人 |
| 审批通过/驳回 | 对待办流程做决策 | 审批人 |

## 7. 视觉风格建议

### 7.1 两种模式

| 模式 | 使用场景 | 风格 |
| --- | --- | --- |
| 深色指挥台 | 演示、大屏、风险态势、智能运营中心 | 深色背景、蓝绿色线条、红橙风险强调 |
| 浅色业务台 | 日常办公、审批、表单、配置后台 | 白色背景、Ant Design 企业风格、信息密度更高 |

建议：

- “智能运营指挥台”可以使用深色，以增强 Palantir / Gotham 式态势感。
- “平台配置中心”和日常表单页面保持浅色企业风格，降低长期使用疲劳。

### 7.2 交互重点

- 风险事件用红色或橙色标记。
- 受影响对象用橙色标记。
- 正常关联对象用蓝色或灰色。
- 已处理节点用绿色。
- 动作按钮使用业务语义，如“生成整改任务”，不要只写“新增”。
- AI 面板必须给出证据、影响范围、建议动作，避免只输出泛泛文字。

## 9. MVP 实现顺序

第一阶段不要做大而全，先完成一个能讲清楚的闭环。

1. 新增“智能运营指挥台”页面。
2. 左侧显示风险事件队列，先内置“3 号产线质量异常”。
3. 中间展示制造业影响图谱。
4. 右侧展示 AI 分析、建议动作和业务按钮。
5. 底部展示处理时间轴。
6. 点击“生成整改任务”后，弹出 CAPA 表单并自动填入上下文。
7. 提交后创建流程/通知记录。
8. 管理员后台保留低代码配置入口，用于解释这些对象、页面、规则和流程可以配置生成。

## 10. 演示话术

推荐演示时不要先讲低代码，而是先讲事故处理故事：

> 假设今天 3 号产线产品缺陷率突然升高。传统方式需要质量、生产、采购、仓库几个部门分别查表、问人、拉群确认。现在系统自动把相关数据连起来，告诉我们问题可能来自哪里，影响了哪些工单、物料、设备和客户订单，并且可以一键生成整改任务。

演示结束再补充：

> 这些业务对象、页面、关系、流程和规则，背后都可以通过平台配置中心低代码配置出来。

这样用户先看到业务价值，再理解平台能力。

## 11. 外部参考

- Palantir Workshop 强调基于 Object layer 构建交互应用，并通过 Actions 和 Functions 支持业务交互。
- Palantir Actions 的公开文档强调用业务动作创建、编辑、删除、链接对象数据，并可在 Workshop 等应用里作为按钮或表单暴露给用户。
- Gotham 类界面适合参考“事件队列 + 中央态势画布 + 右侧方案动作 + 底部时间轴”的指挥调度布局，但制造业系统应把军事地图替换为业务图谱、供应链路径、产线拓扑和订单影响链路。
